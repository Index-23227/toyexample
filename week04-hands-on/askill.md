## 1. 검증 방식

### 왜 고정 수치가 아니라 로직 검증인가

| | 고정 수치 대조 | 로직 검증 (현재 방식) |
|--|----------------|----------------------|
| 테스트 데이터 | 경비내역.xlsx 원본 15건 고정 | **매 실행마다 랜덤 30건 생성** |
| 검증 대상 | "영업1팀 합계 = 362,000인가?" | **"부서별 합계 = 원본 데이터 합산과 일치하는가?"** |
| 새 부서/항목 | 검증 불가 | **개발팀, 복리후생비 등 랜덤 포함하여 검증** |
| 재현성 | 항상 같은 결과 | 매번 다른 데이터로 규칙 검증 (시드 미고정) |

### 검증 흐름

```
[1] 랜덤 데이터 생성
    ├── 경비: 30건 (부서 5종, 항목 6종, 결제방법 2종 랜덤 조합)
    └── 법인: 5개 법인 × 3개월 (금액 랜덤)
         ↓
[2] 자동화 코드 실행
    ├── expense_report.py → [재경]경비정리_test.xlsx
    └── merge_sales.py    → 법인매출통합.xlsx
         ↓
[3] 로직 검증 (28개 항목)
    ├── 규칙이 지켜졌는가? (비고 우선순위, 경비구분 등)
    ├── 시트 구조가 맞는가? (순서, 헤더, 컬럼)
    ├── 수치가 정합한가? (크로스체크: 총괄표 = 전체내역 = 입력)
    └── 정렬이 맞는가? (부서 고정순서, 법인 가나다순)
         ↓
[4] PASS / FAIL 판정
```

### 테스트 데이터 풀

검증 시 사용하는 랜덤 데이터 풀 (기존 3부서/4항목보다 확장):

| 종류 | 기존 (실 데이터) | 검증용 풀 | 목적 |
|------|-----------------|-----------|------|
| 부서 | 영업1팀, 영업2팀, 관리팀 | + **개발팀, 인사팀** | 새 부서 등장 시 처리 검증 |
| 항목 | 교통비, 식대, 사무용품, 접대비 | + **복리후생비, 도서구입비** | 새 항목 → 일반경비 기본 분류 검증 |
| 결제방법 | 법인카드, 개인카드 | 동일 | — |
| 법인 | 5개 | 동일 | — |

## 6. 엣지 케이스 시나리오

현재 `verify_all.py`의 랜덤 데이터 풀이 자연스럽게 커버하는 것과, 향후 추가해야 할 것.

### 이미 커버되는 것 (랜덤 풀에 포함)

| 시나리오 | 커버 방식 |
|----------|-----------|
| 새 부서 등장 (개발팀, 인사팀) | 랜덤 풀에 포함 → T1-07에서 정렬 검증 |
| 새 경비 항목 (복리후생비, 도서구입비) | 랜덤 풀에 포함 → T1-02에서 일반경비 분류 검증 |
| 접대비가 0건인 경우 | 랜덤이므로 확률적으로 발생 → 특별경비 0으로 처리 |
| 개인카드가 0건인 경우 | 랜덤이므로 확률적으로 발생 → 빈 정산 시트 |
| 비고가 전부 None인 법인 | 랜덤 비고 생성에서 75%가 None |

### 향후 추가 권장

| 시나리오 | 위험도 | 추가 방법 |
|----------|--------|-----------|
| 동일 일자 다건 (정렬 안정성) | 낮음 | 같은 날짜 5건 이상 강제 삽입 |
| 금액 0원 | 낮음 | 금액 풀에 0 추가 |
| 법인 파일 3개만 존재 (5개 미만) | 높음 | 랜덤으로 일부 파일 생성 스킵 |
| 법인 파일에 빈 행 | 중간 | 원본에 None 행 삽입 |
| 동일 법인·동일 월 중복 행 | 높음 | 한 법인에 같은 월 2건 삽입 |
| 결제방법이 `체크카드` 등 제3값 | 중간 | PAYMENTS 풀에 추가 |
